应用分类

类型功能需求非功能需求关键点

News Feed

  • Twitter
  • Instagram
  • TikTok
  • Pinterest
  • RSS 类系统
  • 发布图文，视频
  • 关注，被关注
  • 浏览信息流
  • Read low latency：阅读信息流低延迟
  • Consistency：用户关注关系需要在多设备同步。信息流在多设备中顺序保持一致
  • High availability：一般性要求
  • 根据用户粉丝数量制定混合策略：Pull, Push, Fanout（1）
  • 使用指针和日志追加型数据库记录多设备读取位置
  • Read-Your-Writes Consistency（1, 2）
  • 从用户 ID 或者数据 ID 维度进行数据分区（1, 2, 3）

Share

  • TinyURL
  • Bitly
  • Shorby
  • Pastebin
  • 对于给定的 URL 可以生成唯一的短 URL
  • 可以对短 URL 进行定制化，过期设置，分享
  • Read and Write low latency：生成 URL，URL 重定向时低延迟
  • Consistency：每个 URL 对应唯一的短 URL
  • Security：URL 无法被猜测，无法访问权限外的 URL
  • High availability：一般性要求
  • 预生成短 URL 并存储在消息队列
  • 使用哈希表对应 URL
  • 使用 Bloom filter 等方式进行限流（1）

GEO Base

  • Uber
  • Uber Eats
  • Grab
  • Lyft
  • 用户预约车/食物，匹配对应的司机/商家
  • 显示用户和司机实时位置
  • 费用计算以及支付
  • 高匹配度：匹配最合适的服务商（司机，骑手）
  • Consistency：同一个订单只会被最合适的一位服务商接到
  • High availability：一般性要求
  • 使用 Geohash 算法，S2 算法提高匹配度（1, 2）
  • 后端定期更新匹配排名信息
  • 特定应用协议记录司机实时位置
  • 使用日志型数据库记录事件方便数据溯源

IM

  • WhatsApp
  • Facebook Messenger
  • WeChat
  • LINE
  • 从个人/群组中发送/接受信息
  • 消息状态（已送达，已读）和用户状态（在线，离线）
  • 查询历史消息
  • Read and Write low latency：收发信息的速度是关键指标
  • Consistency：保证消息顺序以及在多端的消息同步
  • Security：信息传输安全
  • High availability：一般性要求
  • 特定的应用协议：XMPP, MQTT, AMQP
  • 消息有序性
  • 使用指针和日志追加型数据库记录多设备读取位置
  • 使用点对点加密与安全协议保证信息安全

Live

  • Facebook Live
  • Twitch
  • Zoom
  • Google Meets
  • 浏览音视频直播
  • 发起音视频直播
  • 对直播评论，点赞
  • Read and Write low latency：观看以及发起直播的流畅性
  • Scalability：支撑大量直播以及热门直播
  • Security：只能访问权限内的视频，浏览过程的数据安全
  • High availability：一般性要求
  • 特定的应用协议进行网络自适应：RTMPS, WebRTC
  • 资源自动扩容与收缩支持热门直播
  • 使用多级缓存或者队列服务解决惊群效应

Video

  • YouTube
  • Netflix
  • Disney Plus
  • HBO Max
  • 浏览视频
  • 发布视频
  • 对视频评论，点赞
  • Read low latency: 观看视频的流畅性
  • Scalability：支撑大量视频上传以及观看
  • High availability：一般性要求
  • 边缘节点与 CDN 内容分发
  • 特定应用协议保证网络码率自适应
  • 热门视频使用多级缓存，缓存预热，缓存淘汰策略
  • 资源自动扩容与收缩支持视频处理
  • 工作服务器可以使用 Serverless，Map Reduce 等策略进行视频处理

File Sync

  • Google Drive
  • Dropbox
  • Sync.Com
  • iDrive
  • 上传，更新及下载文件（包括离线状态）
  • 文件共享
  • 多客户端同步
  • Read and Write low latency：上传，更新及下载文件的速度
  • Consistency：多客户端同步文件，保证文件一致性
  • Scalability：文件大数量多的情况下保持高效
  • 使用 Rsync 算法对文件进行增量更新
  • 使用异步上传，分片上传等方式提高更新效率
  • 文件更新可能出现循环依赖

Search

  • Google Search
  • Bing
  • Airbnb Search
  • Twitter Search
  • 搜索文字，显示搜索建议
  • 针对特定类别进行搜索（图片，地点，视频，新闻）
  • Low Latency: 搜索以及搜索建议时保证低延迟
  • High Availability: 确保整个系统的高可用
  • High Scalability: 确保整个系统的高扩展性
  • 匹配：使用 Inverted Index 存储资源
  • 排名：使用 NLP 语义分析，个性化匹配，PageRank 对资源进行排序
  • 缓存热门搜索结果
  • 使用图数据库保存资源关系

Ranking

  • Twitter Top Hashtags
  • Spotify Top K Music
  • 显示排名
  • 排名动态更新
  • Read low latency：显示排名低延迟
  • Scalability：大量数据下高效显示排名以及动态更新
  • 对排名进行预计算，根据业务时间要求更新
  • 对大量更新事务进行合并，改为单一更新事务
  • 使用 Lossy Counting Method，Buckets with Global Query 近似统计

Trading

  • Robinhood
  • Webull
  • TradeStation
  • 浏览股票行情
  • 股票买卖
  • Read and Write low latency：股票行情更新以及股票买卖低延迟
  • Consistency：股票买卖时提供强一致性
  • High availability：一般性要求
  • 强一致性：使用消息队列，乐观锁与悲观锁
  • 保持操作幂等，提供事务回滚

Cooperative editing

  • Google Docs
  • Microsoft Office Online
  • Quip
  • 多人编辑，评论文档
  • 版本回滚
  • 文档权限管理
  • Read and Write low latency：阅读以及编辑文档低延迟
  • Consistency：文档编辑要求强一致性
  • High availability：一般性要求
  • Operational transformation
  • Making collaboration fast

Booking

  • Google Calendar
  • Ticketmaster
  • StubHub
  • 设置空闲时间
  • 预约空闲时间以及更新预约
  • Consistency：预约要求强一致性（某个时间段同时只能被预约一次）
  • High availability：一般性要求
  • Write Skew 和 Phantoms
  • 分布式事务：XA 协议，Seata

Shopping

  • Amazon
  • eBay
  • Etsy
  • AliExpress
  • 浏览商品
  • 购买商品
  • 获取商品状态（发货状态，物流状态）
  • Consistency：避免重复下单，避免商品超卖少卖
  • Security：购物与支付过程的隐私安全以及数据安全
  • High availability：大流量多用户下的高可用
  • 强一致性：使用消息队列，乐观锁与悲观锁
  • 保持操作幂等，提供事务回滚
  • 服务降级与限流

CPU-bound

  • LeetCode
  • Video Editor
  • 进行计算密集型任务处理
  • 获得处理结果以及历史记录
  • 尽快返回结果：对任务进行分块处理，边处理边返回已处理的结果
  • Scalability：支撑大量任务同时高效处理
  • 使用 Serverless 减少系统复杂度
    • 使用 divide and conquer ，map reduce 策略将任务进行分解
  • 异步处理任务，完成后再进行通知

Infrastructure

  • Message queue
  • Object Storage
  • Logging
  • Monitoring
  • 提供基础功能给业务系统如存储，消息推送，监控
  • Scalability：容易使用以及扩展，支撑大量请求
  • High availability：基础组件要求尽量高可用
  • 冗余备份（replication）以及快速恢复（failover）
