设计模式

本文精选了 Azure Cloud Design Patterns 中常用以及与面试相关的模式，重新进行分类以及列出关键点：

Cache-Aside pattern​

使用缓存组件对数据进行分级存储，提高读取速度

何时使用

应用功能需要关注
Video使用 CDN 和 POP 缓存热门视频热门视频提前预热，定时更新缓存视频
Ranking对排名结果进行缓存定期计算排名并且更新缓存
Search对搜索结果进行缓存定期计算搜索结果并且更新缓存

需要关注

  • Issues and considerations
  • 缓存的时效性
  • 删除缓存
  • 缓存预热
  • 缓存一致性
  • 本地缓存与分布式缓存

如何执行

读取数据：

  1. 检查缓存是否有数据，有的话从缓存返回
  2. 如果没有的话，从持久化存储中读取数据
  3. 将数据写入缓存

写数据：

  1. 将数据写入持久化存储
  2. 从缓存删除数据

  • Ease of management via the AWS Management Console

Choreography pattern​

使用异步消息队列对业务进行解耦，兼容多种编程语言与框架，解决业务组件需要单独管理，重试的问题，适用于业务组件经常增删和修改的场景。

何时使用

应用功能需要关注
News Feed发送通知到消息订阅者消息有序性，重试策略
Shopping发送事件通知到业务组件（订单状态，物流状态等）消息持久化，数据回溯
Live发送业务数据到业务组件（评论，点赞）消息有序性，消息持久化

需要关注

  • Issues and considerations
  • 业务组件互相依赖
  • 业务组件过多
  • 使用消息队列的潜在问题

如何执行

使用消息队列组件，当有新的消息则由发布者推送给订阅者。推荐浏览：Cloud Pub/Sub Make Easy

相关组件

名称特点阅读
RabbitMQ
  • Asynchronous Messaging
  • cross-language messaging with multiple languages
  • high availability and throughput
RabbitMQ Features
Pub/Sub
  • At-least-once delivery
  • Exactly-once processing
  • Seek and replay
Pub/Sub All features
Amazon SQS
  • Unlimited Throughput
  • At-Least-Once Delivery
  • Best-Effort Ordering
Amazon SQS features

Retry pattern & Circuit Breaker pattern​

使用代理策略对重试功能进行管理和限制，防止出现大量重试导致原本崩溃的组件无法恢复以及耗用过多重试资源。

何时使用

应用功能需要关注
News Feed对消息发送失败进行重试Exception Handling, Logging
IM对消息发送失败进行重试Exception Handling, Logging
File Sync对业务操作进行重试（更新文件）Retry after delay

需要关注

  • Issues and considerations
  • Exception Handling
  • Logging
  • Recoverability
  • Concurrency
